# Figma component composition from React source — working notes

What we've learned, hypothesized, and validated about generating Figma components from shadcn-style React + Tailwind source. These notes apply to **any** compound component generated by the figma-repo-sync-plugin — not Select-specific. Updated as the architecture solidifies.

---

## The two-layer composition problem

Generating a Figma component from a compound React file (a file like `select.tsx` exporting `Select`, `SelectTrigger`, `SelectContent`, etc.) breaks into two distinct problems that need different solutions:

1. **Outer composition (parent assembly)** — how the sub-components nest inside the parent (`<Select><SelectTrigger /><SelectContent /></Select>`). This is *not* derivable from the compound file itself, because in shadcn the parent is usually `Primitive.Root` re-exported with no visible composition. The composition lives at consumer sites — most reliably in the **colocated story file's first `render` arrow**.
2. **Inner anatomy (per sub-component)** — how each sub-component itself looks: its frame shape (from `cn()`), its slot layout (`{children}`), its decorations (icons, separators). This *is* derivable from each sub-component's JSX body, but requires walking that body — not just reading the `cn()` base classes.

**Confusing them is the common mistake.** Stuffing the story tree's full hierarchy (Select containing Trigger containing Value) into the outer parent doesn't work because Figma INSTANCE nodes can't have arbitrary children appended (see "Figma constraints" below). The story drives outer composition only; sub-component anatomy comes from each sub's own JSX.

---

## Convention: where stories live and what counts as canonical

Centric-ui (and any shadcn-derived library following the convention) colocates stories alongside components:

```
app/components/ui/select.tsx           ← compound source
app/components/ui/select.stories.tsx   ← canonical assembly
```

Inside the story file:
- The first export with a `render: () => (<JSX>)` arrow is treated as canonical.
- Naming convention: `Default`, `Basic`, etc. — but the parser doesn't depend on the name; it depends on order.
- Story files using only `args:` (no `render`) are treated as non-compositional — single-component file conventions still apply.

This convention applies repo-wide: 22/22 component story files in centric-ui follow it as of 2026-05-07.

## What shadcn does and doesn't give us

- **Code-first, no Figma kits.** shadcn is intentionally headless and ships no official Figma guidance. Community Figma kits exist but use varying conventions.
- **Common community convention** (used by most shadcn Figma kits and Tailwind UI alike): the Figma `Select` component represents the **closed/trigger state only**. The popover (`SelectContent` + items) is either (a) a separate component (`SelectMenu`), or (b) a state variant on `Select` (`State: Closed | Open`). Stuffing both into one frame doesn't represent any real visible state — it just looks chaotic.
- **What this means architecturally:** the Figma component for a compound is *not* a literal mirror of the JSX tree. It's a representation of one user-visible state. Multi-state coverage is a separate problem (variants, or sibling components).

---

## Figma constraints that shape the architecture

These are immutable; the architecture must work around them.

1. **An `INSTANCE` node's tree is fixed by its source `COMPONENT`.** You can override individual inner properties (text, fills, visibility) on an instance, but you can't append children. To make `<SelectTrigger><SelectValue /></SelectTrigger>` show up inside an instance, the source `SelectTrigger` Component must be authored with `SelectValue` (or a slot) as a child.
2. **Auto-layout is the right primitive for these components.** Hugs/fills, padding, gaps, and direction are first-class and bind well to `flex` Tailwind classes.
3. **Variables can drive most layout fields** (height, padding, gap, corner radius, font size) and color paints. Modes are the right primitive for axis-driven design tokens (size scales, color themes).
4. **Component properties** (BOOLEAN, TEXT, INSTANCE_SWAP, VARIANT) are flat per Component; they aren't inherited from primitives or HOCs. Each generated Component has to declare its own.
5. **Variant requires `ComponentSet` wrapping.** A standalone Component can't have a VARIANT property — only ComponentSets can. Decisions like "Select has open/closed states" require choosing between independent Components, a ComponentSet wrapper, or an INSTANCE_SWAP between two Components.

## Three-pass compound generation (current working architecture)

For a compound file `app/components/ui/<name>.tsx`:

1. **Pass 1 — generate sub-Components.** For each PascalCase declaration with a `cn(...)` call, create one independent Figma `COMPONENT` keyed by `(sourcePath, componentName)` plugin data. Apply base classes via the styling stub. Add component properties for resolved props (BOOLEAN, TEXT). VARIANT/INSTANCE_SWAP/UNSUPPORTED skipped in v1.
2. **Pass 2 — intra-file composition.** For each sub-component, walk its JSX body for local PascalCase refs (other sub-components in the same file). Place INSTANCE children inside, keyed by plugin-data marker so re-runs don't accumulate. In shadcn this pass usually finds nothing (sub-components rarely reference siblings; they reference primitives) — but it's structurally correct and useful when a sub *does* embed another (e.g., `SelectScrollUpButton` inside `SelectContent`).
3. **Pass 3 — story-driven parent assembly.** Fetch the colocated story, parse the first `render` arrow's JSX tree, find the root element matching `parentName`, place its outermost-level local refs as INSTANCE children of a top-level parent Component. Lowercase HTML tags and member-expression tags are skipped (no Figma analog yet).

Misses are diagnostic, not silent: the LOG panel surfaces `story fetch failed`, `story parsed`, `parent assembled (placed=[…], missing=[…])`.

## Open architectural problem: per-sub-component anatomy from JSX body

**The current gap.** Sub-components are generated as labeled placeholder frames sized by their wrapper's `cn()` classes. They don't visually resemble the real components — `SelectTrigger` should be a button-shaped frame with a chevron icon; `SelectContent` should be a popover with shadow and padded items; `SelectItem` should be a 32px row with an optional check indicator.

**Source data we have.** Each sub-component's full JSX body, e.g.:

```tsx
const SelectTrigger = React.forwardRef<…>(({ className, children, ...props }, ref) => (
  <SelectPrimitive.Trigger
    ref={ref}
    className={cn("flex h-9 w-full items-center justify-between rounded-md border …", className)}
    {...props}
  >
    {children}
    <SelectPrimitive.Icon asChild>
      <ChevronDown className="h-4 w-4 opacity-50" />
    </SelectPrimitive.Icon>
  </SelectPrimitive.Trigger>
))
```

**What needs to happen.** Walk the body, distinguishing:
- The **wrapper element** (usually `Primitive.X` like `SelectPrimitive.Trigger`) — its `cn()` classes drive the outer frame's auto-layout, padding, border, etc. Already handled by styleStub.
- **`{children}` JSX expression** — represents a slot. Should map to either a placeholder frame + INSTANCE_SWAP property, or a labeled "children" slot frame.
- **JSX elements with imported icon names** (`ChevronDown`, `ChevronUp`, `Check`, etc. — typically `lucide-react`) — should map to Figma vector/icon nodes, sized by their `cn()` classes (`h-4 w-4`).
- **Nested external primitives** (`<SelectPrimitive.Icon>`, `<SelectPrimitive.ItemIndicator>`) — usually transparent wrappers; flatten or skip the wrapper, render its children.
- **Other JSX elements** with their own `cn()` — recurse, applying classes to a Figma frame.
- **Spread props, refs, expressions** — skip for visual purposes.

**Decisions needed before implementation:**
- Icon library convention. Lucide is the default for shadcn — generate Figma vectors for known names? Reference an external icon font/library? For centric-ui specifically: `centricsymbols` is the icon font.
- Slot representation. INSTANCE_SWAP requires a default-component reference; v1 of `generateStaticSubComponent` skips it for that reason. Either pick a default empty-component convention or use BOOLEAN show/hide on a labeled placeholder.
- How aggressively to flatten primitive wrappers. `SelectPrimitive.Icon` adds no visual layout; flattening is correct. But heuristics for "transparent wrapper" need to be rigorous (check for absence of `cn()` and presence of `asChild`?).

## Variants vs. independent Components vs. INSTANCE_SWAP

For multi-state representation (open/closed, hover, disabled, error), three Figma patterns are available; pick consciously:

| Pattern | When it fits | Cost |
|---|---|---|
| **VARIANT property on a ComponentSet** | States that share most of their anatomy and differ in fills/strokes/visibility (e.g., button hover/disabled). Designers swap via property. | Requires wrapping all variant cells into one ComponentSet. Variant matrices grow combinatorially. |
| **Independent Components** | States with substantially different anatomy (e.g., `Select` closed = button shape; `Select` open = popover). Designers pick which Component to drag in. | More items in the asset library. Less guidance about which to use when. |
| **INSTANCE_SWAP** | A fixed slot whose contents change but parent layout doesn't (e.g., the icon inside a Button). | Requires a default Component reference; instance-side only allows swapping, not customizing. |

**Default lean for shadcn compounds:** VARIANT for sub-component states (button hover, item selected); independent Components for parent state (`Select` closed vs. `SelectMenu` open); INSTANCE_SWAP for icon slots.

---

## What we explicitly do NOT model (yet)

- **Real-time interaction states.** Figma can show a hover variant; it can't drive it on actual input without prototyping. v1 represents states statically.
- **Animation / transitions.** Out of scope; Figma's animation primitives are too different from CSS.
- **Theme switching at the Figma level.** Tailwind dark mode → Figma color modes works for tokens but doesn't switch a whole component's anatomy. Themed anatomy is rare in shadcn.
- **Responsive breakpoints.** Most shadcn primitives are responsive via parent constraints, not via internal media queries. Figma constraints + auto-layout cover this naturally.

## Patterns confirmed in the 2026-05-08 full-library audit

A retroactive cross-check across 24 generated centric-ui sections (see
[`figma-source-audit-patterns.md`](figma-source-audit-patterns.md) for the
gap inventory). Patterns that stuck:

- **Five source shapes recur** — pure alias, passthrough wrapper, single
  styled wrapper, CVA, compound. The compound rule "≥ 2 PascalCase decls
  with arrow bodies" (regardless of `cn()` or shared prefix) catches every
  shape we've seen — Collapsible (passthrough), ScrollArea/ScrollBar
  (non-prefixed siblings), Select (canonical compound), Card / Dialog /
  Sheet / Sidebar / Form (multi-sub compounds).
- **Pure aliases must be filtered** by body presence. Doing it by `cn()`
  presence (the original rule) was overly restrictive; doing it by body
  presence is correct.
- **`*Indicator` promotion to a private `.SubComponentIndicator`** with
  hidden-by-default instance is the right model. Confirmed across Select
  (ItemIndicator) and Checkbox (Indicator). Generalises to Radio, Menu,
  ToggleGroup when those land.
- **Story-driven slot defaults work for `render: () => …` stories** but
  miss `args:`-only stories (Collapsible, EmptyState). Walker enhancement
  to pull `args:` defaults is the next composition win.
- **Render-prop-becomes-wrapper semantics** (the `<X render={<Y/>}>kids</X>`
  → `<Y kids/>` rewrite) was the right fix. Without it, ItemIndicator's
  span wrapper got separated from its CheckIcon.
- **Section-per-compound + horizontal layout** is the right page
  organisation. Designers can scan by section name; sections sort
  alphabetically; new compounds slot to the right without disturbing
  existing ones.
- **Horizontal section stacking is the rule for ALL multi-section pages**,
  not just the Components page. Sections grow tall internally
  (anatomies pile up vertically; icon grids, spec frames, state matrices
  all extend downward). Stacking sections vertically too forces nested
  scrolling and makes the page unscannable. Side-by-side sections at
  `y = 0` let the canvas read like a bookshelf — pan right to find a
  section, scroll down inside it. Confirmed when the Material Symbols
  icon library was first generated vertically and the user immediately
  asked it be flipped to match the Components page convention.

## Open architectural problem revisited

The "what should `Select` represent in Figma?" question from the original
entry remains open. The audit confirms the same is true for **every other
multi-state component** in the library: Button, Checkbox, Switch, Input,
Tabs, etc. all have an interactive state surface that the current
generation doesn't expose.

Two paths to consider for Phase 4:

- **`Component States` collection per component** (modes-as-variants
  extended to states): one `Button States` collection with modes
  `Default`, `Hover`, `Focus`, `Active`, `Disabled`, `Loading`. Token
  bindings shift per mode. Component structure stays single. Pros: lean
  variant matrix, prototyping maps cleanly to mode switching on hover/
  press. Cons: structural states (loading swaps label for spinner) can't
  be expressed through modes alone.
- **Native VARIANTS for state** with one ComponentSet wrapping
  `State=Default | Hover | Focus | …`. Pros: structural changes possible,
  designers see state surface in the Component panel. Cons: combinatorial
  explosion when combined with other axes.

The decision is the same one Sean settled for variants generally:
**component-specific collections in monolith mode, prefixed `Button —
States`**. Native VARIANTS only when structural changes demand them
(loading-with-spinner). State machinery should be a Phase 4 deliverable.
